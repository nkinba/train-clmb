1. 개요와 핵심 가치

Breakteau는 1인 사용자의 클라이밍 트레이닝(행보드 / 등반 / 보조 근력 / 캠퍼스)을 모바일 우선 PWA로 기록·분석하는 트래커입니다. 궁극 목표는 리드 5.12, 볼더링 V7 달성이며, 부상 방지(통증 0–3 척도) + 폼 코칭용 미디어 첨부가 핵심 차별점.

대상 사용자

단일 사용자 (본인)

다중 사용자/팀 기능은 명시적 out of scope. 모든 결정이 1인 가정에 맞춰져 있어 over-engineering 회피.

플랫폼

모바일 우선 PWA (Android/iOS)

실내 짐 + 야외 암장 환경에서 한 손 조작 + 화면 꺼짐 방지 + 오프라인 입력 우선.

차별점

통증 + 미디어 + 분석

일반 운동 앱이 부족한 부상 추적(어깨/손가락 0-3) + 본인 폼 영상 첨부 + 주간 볼륨/그레이드 추세.

제품 가설 (검증/미검증)

  • 검증 모바일 단일 사용자 — prototype 동안 본인이 직접 사용
  • 검증 PocketBase + 정적 export 조합으로 인프라 비용 ≈ 0
  • 부분 검증 폼 코칭 영상 사용성 — v1.0에서 실 사용 후 재평가
  • 미검증 AI 폼 분석 모델 연동 (v1.2+ 후보)

카탈로그 컬렉션 (S22)

gyms, targets는 picker UI의 데이터 소스. 인증 사용자 read-only + admin만 create/update/delete. 운영 중 PB Admin UI로 항목 추가 가능 → 5분 staleTime 후 picker에 반영.

file 필드 동작 (S18 학습)

  • SQLite엔 파일명 문자열만 저장, 바이너리는 GCS
  • 업로드는 PB가 multipart 청크 buffer로 받음 → e2-micro RAM 1GB에서도 50MB 영상 안전
  • 백업 zip엔 미디어 비포함 → SQLite + 설정만 압축, 가벼움 (현재 ~242KB)

6. 핵심 결정 (ADR)과 트레이드오프

ADR결정핵심 트레이드오프재평가 시점
ADR-1 PocketBase 단일 바이너리 Supabase 대비 자체 운영 부담 ↑ but 외부 종속 ↓, 비용 ↓ 다중 사용자 / 1000+ daily user
ADR-2 외부 종속 최소화 + IaC 학습 모든 다른 결정의 메타 가이드 —
ADR-3 VM Caddy 정적 서빙 (← CF Pages 변경) edge CDN 손실 vs 단일 인프라 운영성 한국 사용자 LCP > 2s 지속
ADR-4 PWA + IndexedDB 오프라인 큐 SW 캐시 일관성 vs 오프라인 입력 가능 —
ADR-5 1인 인증 + localStorage 토큰 + JWT XSS 노출 vs 정적 export 호환 다중 사용자 도입 시
ADR-6 GCS (3차 갱신: GCS→R2→GCS) R2 egress 무료 손실 vs 운용 가능성 egress > 50GB/월 또는 R2 콘솔 안정
ADR-7 Wake Lock + Audio + Vibration + Notification 모바일 브라우저 차이 ↑ vs 타이머 신뢰성 —
ADR-8 GCP us-west1 한국 RTT ~150ms vs Always Free 한도 한국 사용자 RTT > 250ms 지속

결정 변경 패턴 (prototype 회고)

ADR-3 (CF Pages → VM Caddy)와 ADR-6 (GCS → R2 → GCS)는 in-place 갱신. v1.0부터는 supersede 패턴으로 변경 — 옛 ADR 본문은 그대로 두고 "Status: Superseded by ADR-M" 추가 + 새 ADR 작성. 추적성을 위해.

보안 레이어

  • SSL/TLS: Caddy 자동 Let's Encrypt + HSTS preload + TLS 1.3
  • CSP: img-src/media-src에 PB origin만 화이트리스트, frame-ancestors 'none'
  • OS Login: IAM 인증 통합, SSH key 회수 즉시 가능
  • HMAC 키 격리: backup SA + media SA 별도, 버킷별 권한 분리
  • 비root 컨테이너: PB는 uid 100, Caddy/Backup도 최소 권한
  • 인터넷 노출 최소: PB/Backup은 internal network, 외부엔 80/443/443udp만
  • .env 보호: root:600 평문 secret, docker만 접근, 사람은 sudo 필요

9. 학습 — 함정과 패턴

prototype 중 만난 비자명한 함정 8가지 (학습 노트 docs/learning/v1.1-deployment-qa.md에 상세). 모두 운영 first-deploy에서 발견 — 로컬 dev에선 미발견.

CSP cross-origin 차단

img-src/media-src

두 hostname 분리에서 PB origin 차단. connect-src는 추가됐지만 img/media 누락. 로컬은 localhost 단일이라 안 드러남.

PB file-token 별도 인증

backup download 403

admin token으로 list는 200이지만 file get은 403. POST /api/files/token으로 별도 JWT 발급 + ?token= 쿼리 필요.

swap 2GB의 진짜 이유

e2-micro 1GB

steady state 500MB + spike(빌드/백업/마이그레이션) 500MB+가 OOM 위험. 평소 swap used ~0이지만 transient 보험료.

OS Login user 변환

email → POSIX

@·.이 POSIX user에 불가 → _로 변환. uid는 GCP가 전역 단일 보장.

.env 600 root + sudo vim

secret 보호

일반 vim은 read 권한 없으면 빈 buffer로 보임 (오해 유발). sudo vim 또는 sudoedit 필수.

rsync Debian 12 미포함

first deploy 실패

ubuntu-latest는 포함이지만 Debian minimal cloud image는 미포함. sudo apt-get install -y rsync.

file 타입은 SQLite에 메타만

PB 동작 이해

바이너리는 GCS, SQLite엔 파일명만. 백업 zip이 가벼움 + 50MB 영상 업로드 시 RAM 안 폭발 (chunk buffer).

git clone 분리 디렉토리

shell escape

git clone url .에서 .이 줄바꿈으로 분리되면 train-clmb/ 서브폴더로 들어감.

공통 패턴: "로컬 dev에선 안 보이고 운영 first-deploy에서 드러나는" 류의 함정. 다음 phase에선 운영과 가까운 staging 환경 (또는 docker-compose.prod로 로컬 simulate)을 도입할만함.
